Docker Deployment

This is the production path with the fewest moving parts. Download the compose file and environment template, set the password, then start the service.

Prepare configuration

Copy the official `docker-compose.yml` and `.env.example`, then set a strong `CCLOAD_PASS` in `.env`.

Start the service

The default port is `8080`; the admin console is under `/web/`, and API proxy endpoints are under `/v1/*` and `/v1beta/*`.

Create access tokens

After logging in to the admin console, create API tokens on the token page. Proxy endpoints return 401 when no token exists; that is the secure default.

bash 
curl -o docker-compose.yml https://raw.githubusercontent.com/caidaoli/ccLoad/master/docker-compose.yml
curl -o .env https://raw.githubusercontent.com/caidaoli/ccLoad/master/.env.example

# Required: set the admin console password
nano .env

docker-compose up -d
docker-compose logs -f ccload
MySQL is not required by default. Start with SQLite for single-instance deployments. Set `CCLOAD_MYSQL` only when you need external persistence, multiple instances or data retention after Hugging Face restarts.

Hugging Face Spaces

Spaces work well for quick public deployments. The operational risk is that `/tmp` is cleared on restart, so long-term usage should use external MySQL and optionally enable the SQLite replica for faster reads.

Space settings

  • Create a Space and choose Docker as the SDK.
  • Use `ghcr.io/caidaoli/ccload:latest` in `Dockerfile`.
  • Add `CCLOAD_PASS` in Secrets; do not put it in public files.
  • Create an API access token in the admin console before production access.

Persistence guidance

  • Temporary testing: set only `CCLOAD_PASS` and use the default SQLite storage.
  • Long-term usage: set `CCLOAD_MYSQL`.
  • High-read workloads: also set `CCLOAD_ENABLE_SQLITE_REPLICA=1`.
  • Control the log restore window with `CCLOAD_SQLITE_LOG_DAYS`.
Dockerfile 
FROM ghcr.io/caidaoli/ccload:latest

ENV PORT=7860
ENV SQLITE_PATH=/tmp/ccload.db
EXPOSE 7860

Source Builds and Binary Runtime

Use source builds for development and customization, and binaries when you do not want to maintain a build environment. Both require `CCLOAD_PASS` before startup.

⚙️

Source build

Builds must enable the `sonic` tag. Use `make build` directly instead of maintaining a forked build script.

bash 
git clone https://github.com/caidaoli/ccLoad.git
cd ccLoad

export CCLOAD_PASS=your_admin_password
make build
./ccload
📦

Binary runtime

Download the artifact for your platform from Releases, make it executable and run it. Pinning a version is easier to roll back than tracking `latest`.

bash 
wget https://github.com/caidaoli/ccLoad/releases/download/v2.x.x/ccload-linux-amd64
chmod +x ccload-linux-amd64

export CCLOAD_PASS=your_admin_password
./ccload-linux-amd64

Post-deployment verification

Verify the process first, then the configuration entry point, then send a real proxy request with a real token. Do not ignore 401 responses; they usually mean tokens have not been configured yet.

bash 
curl -s http://localhost:8080/health
# /public/summary requires admin login token
curl -s http://localhost:8080/public/summary

# Admin console
# http://localhost:8080/web/

# API proxy endpoint examples
# POST http://localhost:8080/v1/messages
# POST http://localhost:8080/v1/chat/completions
# POST http://localhost:8080/v1beta/models/{model}:generateContent
